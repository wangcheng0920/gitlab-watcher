# GitLab Tag Watcher 设计说明

## 背景与目标

本项目用于监听 GitLab 中由 tag 触发或与指定 tag 关联的流水线状态，并在构建结束时，通过本地桌面消息通知提醒用户。当前阶段不进入代码实现，先明确技术方案、边界和后续实现方向，避免后续补充任务时出现目标漂移。

本次确认的设计目标如下：

1. 运行形态为本地单实例监听器
2. 监听对象为指定项目下、指定 tag 关联的流水线
3. 调度方式采用用户提供的 cron 表达式
4. 通知方式采用“设备适配器优先 + 通用兜底”策略
5. 先关注“发现流水线状态变化并在结束时提醒”，暂不扩展到多项目、多订阅源或复杂任务编排
6. Node.js 包管理工具统一使用 `pnpm`

## 方案比较

### 方案 A：cron 轮询 + 设备适配器通知（推荐）

这是当前推荐方案。由本地进程按照配置的 cron 表达式定时执行一次查询流程：读取目标项目和 tag 配置，调用 GitLab 接口获取关联流水线状态，判断是否进入终态，再通过通知适配器发送本地提醒。

优点：

1. 结构简单，适合从空仓库逐步演进
2. 调度规则显式，便于用户自行调整查询频率
3. 允许按设备能力选择更合适的通知实现，而不是被单一库限制

代价：

1. 不是实时事件驱动，通知时效取决于 cron 频率
2. 需要自行处理重复通知与状态缓存

### 方案 B：固定间隔轮询

由程序内部固定使用例如每 30 秒或每 60 秒轮询一次。实现更直接，但灵活性差，后续如果用户希望在工作时段和非工作时段采用不同频率，会很快遇到限制，因此不作为当前首选。

### 方案 C：Webhook 或事件驱动

理论上实时性更好，但需要额外暴露接收端、维护可达性和鉴权。对于“本地监听并通知”这个初始目标来说过重，因此当前不采用。

## 当前确认的通知策略

基于本次讨论，通知层不再按“统一使用 `node-notifier`”理解，而改为“**设备适配器优先，`node-notifier` 兜底**”。

当前已确认的规则如下：

1. macOS 设备优先使用 `osascript`
2. macOS 当前选定的提醒方式为 `display alert`
3. 该提醒属于阻塞式弹窗，用户必须手动关闭
4. 后续如接入新的设备类型，应新增对应适配器，而不是先修改主流程
5. 当运行环境没有专用适配器时，再退回 `node-notifier` 作为通用兜底

这里的关键结论是：当前业务需要的不是普通通知中心消息，而是**会打断用户并要求手动确认的完成提醒**。因此在 macOS 上，不应继续把 `node-notifier` 视为默认首选，也不应使用 `osascript` 的 `display notification`。

## 推荐架构

推荐采用“调度层 + 查询层 + 文件状态层 + 通知层”的四段式结构。

1. **调度层**：负责解析和触发 cron 表达式，在指定时间点启动一次轮询任务。
2. **查询层**：负责与 GitLab API 交互，获取指定项目、指定 tag 以及对应流水线状态。
3. **文件状态层**：负责基于目录迁移和文件追加记录任务进度，不引入数据库。
4. **通知层**：负责将“成功 / 失败 / 取消”等终态转换为用户可读消息，并根据当前设备选择对应通知适配器发送提醒。

这个划分的好处是职责清晰。后续如果从单实例扩展为多监听任务，调度层和文件状态层可以先演进，而查询层与通知层基本可以复用。通知层内部再细分为“通知门面 + 平台适配器”即可，不需要把设备差异扩散到轮询主流程。

## 文件系统方案

当前阶段已确认：**不创建数据库，监听任务完全基于文件系统流转**。

建议目录结构如下：

```text
tasks/
  pending/
    <tag>.md
  processing/
    <tag>.md
  archive/
    success/
      <tag>.md
    failed/
      <tag>.md
    canceled/
      <tag>.md
```

约定如下：

1. `pending/` 下的文件表示待开始任务，初始化时允许直接为空文件。
2. `processing/` 下的文件表示正在监听中的任务，不再额外按状态分目录。
3. `archive/` 只在任务完成后使用，并且仅按流水线终态区分为 `success`、`failed`、`canceled` 三类。
4. 当前范围接受“**一个 tag 只处理一次**”，因此归档文件保持原文件名，不增加时间前缀。
5. 目录位置表达任务的大阶段，文件内容表达轮询历史和最新状态。

## 核心数据流

一次完整轮询建议遵循以下流转：

1. 用户在 `tasks/pending/` 下创建一个以 tag 命名的空文件，表示新增监听任务。
2. 调度器读取本地配置，获得 GitLab 项目标识、cron 表达式和通知偏好。
3. cron 触发后，调度器扫描 `pending/` 和 `processing/`。
4. 新任务被接手时，先执行并发保护，再把 `<tag>.md` 从 `pending/` 移动到 `processing/`。
5. 在 `processing/` 中调用 GitLab 查询接口，定位与目标 tag 关联的目标流水线。
6. 每次查询后，不覆盖整份任务模型，而是向 `<tag>.md` 顶部追加一段最新记录，按时间倒序保留历史。
7. 如果本次状态仍是 `not_found`、`running` 或 `query_error`，文件继续留在 `processing/`。
8. 如果目标流水线首次进入 `success`、`failed` 或 `canceled`，则生成通知内容并触发本地消息。
9. 通知完成后，把 `<tag>.md` 直接移动到 `archive/<status>/`，后续轮询不再处理该文件。

这里的关键不是维护复杂的任务对象，而是让**文件迁移表达阶段，文件内容表达查询历史**。

## 启动恢复与轮询主流程

当前阶段建议程序启动后，先执行一次“未完结任务恢复”，再进入 cron 驱动的持续轮询。

建议主流程如下：

1. 程序启动后先扫描 `tasks/pending/` 和 `tasks/processing/`。
2. 两个目录中的文件都视为“未完结任务来源”。
3. 来自 `pending/` 的任务需要先加锁并移动到 `processing/`，表示任务正式开始被接管。
4. 来自 `processing/` 的任务不做迁移，直接继续处理。
5. 所有进入处理阶段的任务，都统一调用同一套查询与状态写入流程。
6. 程序完成首次恢复扫描后，再按 cron 表达式持续执行同一套任务处理逻辑。

这里建议特别约束两点：

1. **启动扫描和 cron 轮询必须复用同一套处理函数**，避免形成两套状态推进逻辑。
2. **处理单位始终是单文件单任务**，不要先聚合为批量状态对象再统一推进，否则会破坏当前“文件即任务”的简单模型。

## 文件内容格式

当前建议任务文件采用“纯文本分段追加”的极简格式，每次查询都把最新结果写在文件最上方，例如：

```text
---
queryTime: 2026-05-21T14:25:00.000Z
status: success
pipelineId: 102938

---
queryTime: 2026-05-21T14:20:00.000Z
status: running
pipelineId: 102938

---
queryTime: 2026-05-21T14:15:00.000Z
status: not_found
```

这样设计的意义如下：

1. 打开文件时优先看到最新状态。
2. 不需要额外日志文件，也能保留完整轮询历史。
3. `pipelineId` 会随历史一起保留，便于后续实现更稳妥的通知去重判断。

## 通知层设计建议

当前建议通知层采用统一接口，例如 `notify(payload)`，主流程只依赖该接口，不直接依赖具体平台命令或第三方库。

建议职责拆分如下：

1. **通知门面**：接收统一通知载荷，并根据运行环境选择适配器
2. **Mac 适配器**：使用 `osascript` 调用 `display alert`
3. **Fallback 适配器**：在没有专用设备适配器时，退回 `node-notifier`

当前不建议把“mac 判断 + shell 命令拼接 + fallback 判断”直接写进任务处理主流程。那样会把平台差异和业务逻辑耦合在一起，后续新增设备时会不断侵入核心流程。

同时需要明确一点：`display alert` 在当前设计中的职责是“强提醒”，不是“通知中心留痕”。因此后续编码时应围绕“是否阻塞用户、是否需要手动关闭”来验证行为，而不是仅验证通知是否成功发出。

## 错误处理边界

当前设计建议将错误分为三类：

1. **配置错误**：例如缺失项目 ID、tag 或 cron 表达式。这类问题应在启动或加载配置时直接暴露，避免进入轮询流程。
2. **查询错误**：例如 GitLab 接口不可达、鉴权失败、目标 tag 对应流水线不存在。应记录清晰错误上下文，并与“流水线失败”区分开，避免误报。
3. **通知错误**：例如 `osascript` 执行失败、弹窗创建失败或 fallback 通知失败。该错误不应回写为流水线失败，但需要保留可见性，避免用户误以为提醒已经送达。

在当前文件系统方案下，建议进一步明确：

1. `not_found` 表示查询成功但尚未找到目标流水线，不等同于查询失败。
2. `query_error` 表示本次查询异常，任务**不终止**，文件继续保留在 `processing/` 等待下次轮询。
3. 只有流水线终态 `success`、`failed`、`canceled` 才会触发归档。

同时，轮询任务应避免并发重入。即使是单文件方案，也应在扫描或迁移前先加锁；如果上一次查询尚未结束，新的 cron 触发应被拒绝或延后，否则容易造成重复查询和文件写入覆盖。

## 后续实现建议

后续进入实现阶段时，建议先按以下顺序推进：

1. 定义配置文件结构与必填项
2. 明确 GitLab 查询所需的 API 路径和筛选规则
3. 落实基于目录迁移和文件追加的本地状态格式
4. 明确启动恢复、任务扫描和统一处理函数
5. 接入通知门面、macOS `display alert` 适配器与 `node-notifier` fallback
6. 在最后再串联 cron 调度和完整轮询流程

这样可以先稳定“查什么、记什么、何时通知”，再补上调度层，降低联调复杂度。

## 工具与依赖管理约定

当前项目的 Node.js 依赖管理工具统一为 `pnpm`。后续新增依赖、安装依赖、执行脚本以及维护锁文件时，都应优先使用 `pnpm`，避免同时混用 `npm` 与 `pnpm` 产生多套锁文件和依赖树差异。

建议约定如下：

1. 安装依赖使用 `pnpm add`
2. 安装开发依赖使用 `pnpm add -D`
3. 执行脚本使用 `pnpm <script>`
4. 锁文件以 `pnpm-lock.yaml` 为准，不再维护 `package-lock.json`
