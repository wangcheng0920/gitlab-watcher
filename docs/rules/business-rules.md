# GitLab Watcher 业务规则

## 文档目的

本文件用于约束后续 AI 在本仓库中编写业务代码时必须遵守的业务规则。

它不是技术设计草案，也不是可选建议，而是当前阶段的业务边界说明。后续若实现细节与本文件冲突，应优先回到需求澄清，而不是由 AI 自行扩展范围或修改业务语义。

## 使用要求

后续 AI 在开始实现、修改或补充业务代码前，应先阅读本文件，并将其作为业务行为判断依据。

具体要求如下：

1. 不得把设计建议当作已确认业务规则
2. 不得在未明确确认前主动扩大监听范围
3. 不得为了“通用性”引入超出当前业务目标的抽象
4. 若实现方案与本文件冲突，应先澄清，再继续编写代码

## 本文档与 specs 的边界

本文件维护的是“业务规则”，不是“数据规格”。

这里记录的内容应回答以下问题：

1. 系统必须做什么
2. 系统不能做什么
3. 什么行为才符合当前业务语义
4. 什么边界未经确认前不得扩展

因此，适合写在本文件中的内容包括：

1. 监听范围约束
2. 查询语义约束
3. 通知触发条件
4. 重复通知限制
5. 状态变化背后的业务含义

如果要定义的是以下内容，则更适合维护在 `docs/specs/` 中：

1. 数据结构
2. 字段名称
3. 状态枚举值
4. 状态流转关系
5. 示例对象和示例存储格式

可以用一句话区分：

**业务对错看 rules，结构定义看 specs。**

## 业务目标

当前项目的业务目标是：

监听指定 GitLab 项目下、指定 tag 关联的流水线状态，并在该流水线构建完成后，通过本地消息通知提醒使用者。

这里的核心业务价值是“完成提醒”，不是通用流水线平台，也不是多任务调度中心。

## 规则一：监听范围

当前已确认的监听范围必须严格限定为以下边界：

1. 单实例运行
2. 单项目监听
3. 指定 tag 关联流水线监听
4. 本地执行、本地提醒

AI 在实现时必须遵守以下规则：

1. 不得默认扩展为多项目监听
2. 不得默认扩展为多 tag 并行监听
3. 不得默认扩展为多用户、多租户或服务端集中部署
4. 不得在未确认前引入任务池、订阅列表、监听器注册中心等超前设计

如果后续需要扩展为多项目或多监听任务，应作为新的业务需求单独确认，而不是在当前实现中预埋复杂结构。

## 规则二：查询规则

查询行为必须围绕“找到指定项目下、指定 tag 关联的目标流水线，并判断其状态”展开。

AI 在实现请求逻辑时必须遵守以下规则：

1. 查询必须携带 GitLab 所需鉴权信息，当前已确认使用 `PRIVATE-TOKEN`
2. 查询触发方式必须由轮询间隔驱动，并在一轮执行完成后再安排下一轮
3. 轮询间隔属于配置项，当前字段名为 `pollIntervalMinutes`，不得写死为不可修改常量
4. 查询结果应服务于判断流水线状态，不应扩展为无关统计或批量同步

同时必须区分以下几类情况：

1. **查询成功且找到目标流水线**
2. **查询成功但未找到目标流水线**
3. **查询失败**，例如网络错误、鉴权失败或接口异常

这三类情况的业务含义不同，后续实现中不得混为一谈，更不能把“查询失败”误报为“流水线失败”。

## 规则三：通知触发规则

通知的触发条件必须收敛到“目标流水线首次进入终态”。

AI 在实现通知逻辑时必须遵守以下规则：

1. 流水线处于运行中、等待中或其他非终态时，不发送完成通知
2. 流水线进入终态后，才允许触发本地通知
3. 当前至少应覆盖 `success`、`failed`、`canceled` 等终态
4. 通知内容应尽量包含 tag、项目标识和终态结果，保证用户能直接理解提醒含义
5. 当前 macOS 场景下，完成提醒应采用需要用户手动关闭的阻塞式弹窗语义，而不是可自动消失的普通通知

当前阶段的通知目标是“让用户知道这次构建已经结束以及结果是什么”，并在 macOS 当前方案下以阻塞式弹窗强制用户看到提醒，而不是提供完整流水线详情页或复杂交互。

## 规则四：重复通知规则

重复通知控制是当前业务中的硬约束，不是优化项。

AI 在实现状态记录或去重逻辑时必须遵守以下规则：

1. 同一条流水线进入终态后，只应通知一次
2. 不能因为后续轮询再次读到同样结果而重复弹窗
3. 去重依据应围绕流水线唯一标识和已通知状态设计
4. 不得仅使用“上一次是否成功”这类过于粗糙的布尔标记替代正式去重逻辑

如果没有去重能力，系统会在同一终态上重复提醒，直接破坏使用体验，因此该规则在后续实现中必须优先保证。

## 当前明确不在范围内的内容

在未新增需求前，以下内容都不属于当前业务范围：

1. 多项目统一监听
2. 多 tag 批量监听
3. Webhook 回调接入
4. 服务端部署与远程通知分发
5. 复杂权限体系、用户体系或任务编排能力

AI 不得因为“未来可能用到”而提前实现这些内容。

## 规则五：退出与重试规则

当前 watcher 支持两种运行模式，退出语义不同：

### 间隔模式（CLI，`pnpm start`）

1. 启动后应先立即执行一轮处理，再根据未完结任务决定是否安排下一轮
2. 当 `tasks/pending/` 与 `tasks/processing/` 都没有未完结任务时，watcher 应自动退出
3. 终态通知失败不表示任务完成，任务必须保留在 `processing/` 等待后续重试
4. 对于最新记录为 `notify_error` 的任务，后续轮询应直接基于本地已确认终态重试通知，而不是重新查询 GitLab

### 常驻模式（Serve，`pnpm serve`）

1. 启动后应立即执行一轮处理，之后按 `pollIntervalMinutes` 持续轮询
2. 当 `tasks/pending/` 与 `tasks/processing/` 都没有未完结任务时，watcher 不应退出，而是继续等待新任务
3. 退出仅通过 SIGINT/SIGTERM 信号触发
4. 终态通知失败与重试规则同间隔模式

## 文档维护规则

当业务目标、监听范围、通知条件或查询语义发生变化时，应优先更新本文件，再继续后续实现。

若同时存在设计文档和实现代码，业务含义以本文件中已确认规则为准；设计文档用于说明实现方式，不替代业务规则本身。
